# WearBridge Protocol

This document defines the expected protocol between:

- phone app (`io.vibe.wearbridge`)
- watch app (currently WearLoad-compatible)
- `tools/watch_adb_install.sh` (adb automation)

It covers both wire protocol and the expected free-text log format used today.

## 1. Compatibility Target

WearBridge phone-side protocol is compatible with existing WearLoad watch-side paths and keys.

Current implementation uses:

- Wear `MessageClient` paths for control/log/events.
- Wear `DataClient` `DataItem` for APK payloads and APK export asset.
- Phone-side logcat session lines (`tag=WearBridge`) for adb polling.

## 2. Phone -> Watch Protocol

All path constants are defined in `app/src/main/java/io/vibe/wearbridge/protocol/WearProtocol.kt`.

### 2.1 Message paths

- `/request-sync`
  - Payload: empty bytes
  - Meaning: request app list sync

- `/request-apk`
  - Payload: UTF-8 package name
  - Meaning: request APK export from watch

- `/delete-app`
  - Payload: UTF-8 package name
  - Meaning: request uninstall on watch

- `/check-companion`
  - Payload: empty bytes
  - Meaning: request companion version info

### 2.2 Install payload (`DataItem`)

- Path prefix: `/apk/`
- Concrete path generated by phone: `/apk/<epoch_millis>`
- Keys:
  - `package_name`: string
  - `apk_count`: int
  - For each `i` in `[0..apk_count-1]`:
    - `apk_<i>`: `Asset`
    - `file_name_<i>`: string
    - `apk_size_<i>`: long

`queued_to_datalayer` means `putDataItem` completed on phone side only.

## 3. Watch -> Phone Protocol

### 3.1 Message paths

- `/log-message`
  - Payload: UTF-8 free text from watch
  - Phone handling: routed into `BridgeState.logWatchMessage(...)`

- `/app-list-start`
  - Payload: UTF-8 integer (expected chunk count) or empty

- `/app-list-chunk`
  - Payload: JSON array of `RemoteAppInfo`

- `/app-list-end`
  - Payload: empty

- `/check-companion-response`
  - Payload: JSON `CompanionInfo`

### 3.2 Data path

- `/apk-export`
  - Data map keys:
    - `apk_file` (`Asset`)
    - `package_name` (string, optional for file naming)
    - `app_label` (string, optional for file naming)

## 4. Phone App Intents

Main activity: `io.vibe.wearbridge/.MainActivity`

### 4.1 Launcher intent

- `android.intent.action.MAIN` + `android.intent.category.LAUNCHER`

### 4.2 Share intents

- `android.intent.action.SEND`
  - Supported MIME types (manifest):
    - `application/vnd.android.package-archive`
    - `application/octet-stream`
    - `application/zip`
    - `application/x-zip-compressed`
  - URI source priority at runtime:
    - `EXTRA_STREAM`
    - `ClipData`
    - `intent.data` fallback

- `android.intent.action.SEND_MULTIPLE`
  - Supported MIME types (manifest): same as `SEND`
  - URI source priority at runtime:
    - `EXTRA_STREAM` list
    - `ClipData`
    - `intent.data` fallback

### 4.3 View intent

- `android.intent.action.VIEW`
  - Manifest-supported data:
    - scheme `content`, mime `application/vnd.android.package-archive`
    - scheme `file`, mime `application/vnd.android.package-archive`
  - Runtime URI sources:
    - `intent.data`
    - plus `ClipData` if present

### 4.4 Custom install intent (used by adb script)

- Action: `io.vibe.wearbridge.action.INSTALL_TO_WATCH`
- URI source:
  - `io.vibe.wearbridge.extra.FILE_COUNT` + `io.vibe.wearbridge.extra.FILE_URI_<i>`
- Extras:
  - `io.vibe.wearbridge.extra.AUTO_SEND` (bool, defaults to `true` for this action)
  - `io.vibe.wearbridge.extra.PACKAGE_NAME` (optional string override)
  - `io.vibe.wearbridge.extra.SESSION_ID` (string)

For non-custom intents, `AUTO_SEND` default is `false` unless explicit extra is set.

URIs used by `tools/watch_adb_install.sh` are `file://` paths under:

- `/data/user/0/io.vibe.wearbridge/files/WearBridgeStaging/payload/slot-XXXX.apk`

## 5. Session and Logcat Protocol

Phone emits session lines to logcat (`tag=WearBridge`) in this format:

- `session=<session_id> state=<state> [k=v ...]`
- `session=<session_id> watch="<watch_free_text>"`

Main state values currently emitted:

- `session_started`
- `files_selected`
- `package_detected`
- `package_override`
- `package_missing`
- `auto_send_started`
- `upload_progress` (includes `percent=<0..100> stage=<stage> ...`)
- `auto_send_queued`
- `auto_send_failed`
- `processing_failed`
- `awaiting_manual_send`
- `watch_terminal`
- `session_finished reason=<reason>`

`tools/watch_adb_install.sh` polls these session lines from logcat.

Success condition in script:

- `state=session_finished reason=watch_terminal`

Failure conditions in script:

- any other terminal `session_finished reason=...`
- `auto_send_failed` / `processing_failed`
- timeout only when `--poll-seconds N` is explicitly provided

Default poll mode has no timeout and is intended to be stopped by Ctrl-C if needed.

## 6. Expected `/log-message` Free-Text Format

Current watch compatibility is text-based. There is no strict schema yet.

To guarantee phone session closure with current parser, terminal watch message text must contain at least one terminal keyword (case-insensitive):

- `status_success`
- `status_failure`
- `failure`
- `error`
- `ошиб`
- `успеш`
- `aborted`
- `incompatible`
- `storage`

If none of the above appears, session may remain open from script perspective.

### 6.1 Recommended free-text templates (expected)

Use these text lines on `/log-message`:

- Progress (non-terminal):
  - `INSTALL_PROGRESS phase=<phase> file=<name> index=<n>/<total>`

- User action required:
  - `INSTALL_PENDING_USER_ACTION message=<text>`

- Terminal success:
  - `INSTALL_STATUS status_success package=<package> code=0 message=<text>`

- Terminal failure:
  - `INSTALL_STATUS status_failure package=<package> code=<installer_code> message=<text>`

These remain free text but are stable enough for parsing and human logs.

## 7. Example Flow (Install)

1. Script stages APK(s) in fixed app-private slots and launches `ACTION_INSTALL_TO_WATCH`.
2. Phone logs:
   - `session_started`
   - `files_selected`
   - `package_detected`
   - `auto_send_started`
   - `upload_progress ...`
   - `auto_send_queued`
3. Watch receives `/apk/...`, writes assets, commits PackageInstaller session.
4. Watch sends `/log-message` lines.
5. Phone logs each as `session=<id> watch="..."`.
6. If watch terminal text includes terminal keyword, phone logs:
   - `state=watch_terminal`
   - `state=session_finished reason=watch_terminal`
7. Script exits success.
